Overview & Goals

Making the invisible network visible

The P2P Explorer started as a Nethermind initiative to surface the Starknet peer-to-peer network — a layer that existed but had no public face. The result is a live network crawler and explorer that turns raw libp2p data into a human-readable dashboard: where are the nodes, what software are they running, and is the network healthy?

Initially scoped as the P2P MVP (synchronising Juno nodes via a peer-to-peer network), the project evolved into a broader vision: a public-facing explorer that gives the Starknet community a real-time pulse on the base layer.

Three goals, one dashboard

01

Make the network feel real

Provide a global perspective on where Starknet nodes are, how many people support the base layer, and which geographies are most active.

02

Inform upgrade decisions

Let the community see at a glance what software all nodes are running and the global distribution of client versions — critical data for protocol governance.

03

Indicate network health

Measure block propagation latency across the network, how well it responds to reorgs, and the rate at which new blocks reach all nodes.

Network Architecture

Two types of nodes, one network

The Starknet P2P network is built on libp2p and designed around two distinct node roles. Understanding this distinction was foundational to designing both the MVP and the explorer UI — every metric, every status indicator maps back to one of these two roles.

Type 01

Boot Nodes

Discovery nodes that act as the network's entry point. They publish new blocks into the network by gossiping blocks from the Starknet feeder gateway to all peers.

  • Sync only from the feeder gateway
  • Serve blocks to connecting peers
  • Do not sync from other nodes
  • Required for any normal node to join

Type 02

Normal Nodes

Full nodes that sync exclusively from the P2P network. They use libp2p's Kademlia DHT for peer discovery, requiring at least one boot node address to bootstrap.

  • Sync blocks, transactions, and receipts from peers
  • Use Kademlia DHT for discovery
  • Determine sync height from peer information
  • Listen for new blocks gossiped by boot nodes

How sync flows through the network

01

Boot node picks up new block

The sequencer publishes a new block through the feeder gateway. Boot nodes, syncing directly from this gateway, pick it up immediately.

02

Block is gossiped across the network

Boot nodes gossip the new block to all connected peers. Without a consensus mechanism, this gossip protocol is the only path for block propagation.

03

Normal nodes discover peers and sync

Normal nodes use Kademlia DHT to discover peers, negotiate a target height from what the network reports, and pull block data from their peers.

04

Node reaches tip and listens

Once synced, a normal node listens for gossiped new blocks and begins serving data to its own connected peers, contributing to the network.

MVP Scope

What's in the first version

The MVP is intentionally narrow: a network of Juno nodes that can synchronise a shared database using the P2P network and serve RPC requests across all Starknet networks. This is not a multi-client effort yet — Pathfinder and other full nodes are explicitly out of scope for this iteration.

Scope boundaries

In

Juno node sync via P2P

Juno nodes synchronise blocks, transactions, receipts, state diffs, and classes from each other without touching the feeder gateway.

In

RPC on all networks

P2P-synced nodes serve full Starknet RPC requests — mainnet, testnet — passing existing RPC test suites.

In

Boot + Normal node roles

The two-tier node model is fully operational: boot nodes publish, normal nodes discover and sync.

Out

Other client interop

Pathfinder, Deoxys, and Papyrus compatibility is a future milestone. This iteration is Juno-to-Juno only.

Technical Decisions

What we locked in — and what we didn't

The implementation required a series of foundational technical choices, many of which had downstream design implications for how the explorer would surface node behaviour and network state. These decisions also shaped the complexity of the rollout.

Decisions made

Two node types — Boot nodes and normal nodes with distinct roles and sync strategies.

libp2p Kademlia DHT for discovery, chosen over rendezvous protocol for better decentralisation.

Ed25519 key scheme for node identity derivation, aligned with the existing Starknet P2P spec.

Protocol Buffers for message serialisation — typed, efficient, and cross-client compatible.

All block body parts from one peer — instead of fetching different parts from a range of peers, reducing coordination overhead.

CurrentBlockHeaderRequest message added to the spec, enabling nodes to query a peer's current chain tip.

Events added to transaction receipts — a spec extension required for full block reconstruction by normal nodes.

Still to be decided

Block request batching — how many blocks to request per peer: one block per peer, or a range in a single request.

Blacklisting strategy — how misbehaving peers are identified, penalised, and removed from the peer set.

Class chunking — whether class definitions can always be transmitted without chunking, or if large classes need segmentation.

Transaction receipt commitment — currently no way to verify that a receipt was transmitted correctly; verification mechanism TBD.

Single vs multiple protocols — whether all message types share a single protocol ID or each block component has its own.

StateDiff and Block Body message changes — the current spec lacks DeployedContracts and ReplacedClasses; the resolution approach is still under discussion.

Testing Strategy

How we know it works

The testing strategy spans two levels: end-to-end tests that validate real network behaviour with actual Juno nodes, and integration/unit tests built on libp2p's Mocknet for fast, isolated verification of protocol correctness.

E2E — Initial Sync

01

Set up the boot node and sync it to block K from the feeder gateway.

02

Spin up N P2P nodes configured to sync directly from the boot node.

03

Verify all P2P nodes have reached block K and the block data is consistent.

04

Run the full existing RPC test suite against each P2P node to confirm correct data serving.

E2E — Continuous Sync

01

P2P nodes periodically receive updates about joined/disconnected peers and new blocks via gossip.

02

Continuously monitor each node's progress using starknet_blockNumber and assert forward progress.

E2E — Resilience Scenarios

→

All peers resume syncing correctly when a boot node goes down and then recovers.

→

A normal node that disconnects mid-sync can resume from its last known state without re-syncing from genesis.

Integration/Unit — libp2p Mocknet

→

Networks of varying sizes (boot nodes + normal nodes) all reach the same sync height.

→

Nodes disconnect, reconnect, and continue syncing without data loss or stalls.

→

Messages gossip correctly to all nodes in the network within expected latency bounds.

→

Incorrect messages and invalid signatures are rejected and do not propagate through the network.

Status & Roadmap

Where the implementation stands

The P2P work is mid-flight. The foundational layer — discovery, adapters, spec changes, and header sync — is in place. Putting a complete block into the database and supporting live continuous sync are the remaining critical milestones before the MVP is shippable.

Done

  • Necessary P2P spec changes (7 PRs across client teams)
  • Data adapters between P2P and Juno formats
  • Discovery setup via Kademlia DHT
  • Block header syncing (not yet persisted to DB)
  • Yamux multiplexer integration validated

Remaining

  • Put full block data into the database
  • Support continuous sync (live block following)
  • End-to-end and integration test suite
  • Blacklisting for misbehaving nodes
  • Missing transaction fields (reported by community)
  • V3 transaction support
  • StateDiff spec modifications and migration

The network crawler vision

Beyond the MVP sync work, the longer-term goal is a public network crawler — a live map of every Starknet node globally. The data it surfaces directly informs the three goals of the explorer.

Client type

Juno, Pathfinder, Deoxys, Papyrus — distribution across all full-node implementations.

Geographic distribution

Real-time semi-obfuscated global node map. Where is Starknet running?

Client version

Version breakdown across the network — essential for upgrade readiness and governance.

Block propagation latency

How fast does a new block reach every node? Measured with a global relay infrastructure.

Underlying network

AWS, GCP, Hetzner, TOR — tracks infrastructure concentration and network health.

Topology coefficient

A novel graph metric reflecting topological consistency — how well-connected and resilient is the network?

Retrospective

What slowed us down — and what we learned

The P2P MVP surfaced a common trap in protocol-level work: assuming that an early PoC represents the full shape of the problem. The initial implementation obscured how much coordination, rework, and cross-team alignment a production-grade P2P sync actually requires.

The PoC created a false baseline

The initial PoC included informal "hacks" to the P2P spec that gave a misleading picture of the effort required to reach production quality. Timelines were set against that baseline — and were off.

Spec alignment across teams was the real bottleneck

Seven open PRs for P2P spec changes created a continuous overhead of cross-team discussion, review, and rework. Even moving ahead of merged PRs meant staying in sync with live debates. At one point, handler unit tests were intentionally dropped to keep pace with spec changes — a debt that needed repayment later.

Design questions surface late in protocol work

Only once initial sync started materialising did the detailed design questions appear — block batching, blacklisting, StateDiff structure, StateUpdate derivation. These aren't avoidable, but planning buffer for them is essential.

The StateDiff problem had no clean answer

The P2P spec's StateDiff lacked DeployedContracts and ReplacedClasses. Multiple resolution approaches were explored — adding a StateUpdate message, adding old_root to StateDiff, or changing core DB representation. Each had significant trade-offs. The short-term resolution was Option 2 (old_root in StateDiff), deferring the larger structural changes.
